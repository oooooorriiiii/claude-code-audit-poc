Claude Code を
ローカルで監査する

OpenTelemetry で「誰が・何を入力し・何をさせたか」を手元に残す

単一 Laptop + Docker Compose / Claude Code Pro

こんにちは。Claude Code を組織で使うとき「監査ログ、どこまで取れるの?」という話をします。 結論、Pro プランでも自分の Laptop だけで、入力からツール実行まで全部ローカルに残せます。5分で実演します。

課題

  • Claude Code は強力 = 何でもできる(bash・ファイル編集・MCP)
  • 「誰が・何を入力し・どのツールを・どう判断して実行したか」を
    後から追跡したい
  • でも 外部 SaaS に会話を送りたくない → 手元で完結させたい

→ Claude Code の OpenTelemetry 出力をローカルで受けて保存・可視化する

課題はシンプル。Claude Code は bash もファイル編集もできる。だから監査したい。 でも会話を外のサービスに出したくない。手元で完結させたい。 実は Claude Code は OpenTelemetry で利用状況を吐けるので、それをローカルで受けます。

前提: この3つだけ押さえれば OK

種類何を表すか保存先
メトリクス数値の時系列
例: トークン数・コスト・回数 → 「いつ・どれだけ」		Prometheus
ログ/イベント1件1件の出来事
例: 誰が何を入力・どのツール実行 → 「何が起きた」		Loki
トレース1リクエストの処理の流れと所要時間
→ 「どの順で・どこが遅い」		Tempo

これらを吐き出す共通規格が OpenTelemetry (OTel)。Claude Code は標準対応。
※ Zabbix / Datadog / ELK 等を運用した経験があれば「監視データの“送る側”を標準化したもの」と思えば OK

OTel を触ったことがない方向けに、前提だけ。監視データは大きく3種類です。 数値の時系列＝メトリクス、これは Prometheus に。1件1件の出来事＝ログ/イベント、これは Loki に。 1つの処理の流れと所要時間＝トレース、これは Tempo に入れます。 これらを標準フォーマットで吐き出す仕組みが OpenTelemetry。 Zabbix や Datadog、ELK を運用された経験があれば、その“データを送る側”を業界標準にしたもの、と捉えてください。

アーキテクチャ

ホストの claude に環境変数を渡すだけ。受信側(Collector〜Grafana)は docker compose up 一発。

構成はこれだけです。普段ログ基盤を運用されている方なら馴染みのある形だと思います。 左のホストで動く Claude Code が、テレメトリを「OTel Collector」という受け口に送ります(OTLP という共通フォーマット、gRPC の 4317 番)。 Collector はいわば仕分け係で、数値のメトリクスは Prometheus、ログ/イベントは Loki、トレースは Tempo に振り分け、 同時に生 JSON をファイルにも保全します。最後に Grafana で全部まとめて見る。受信側は全部 Docker、compose 一発で立ちます。

デフォルトで取れるもの

Metrics

token.usagecost.usage session.countlines_of_code code_edit.decisionactive_time

Events (logs)

user_prompttool_result tool_decisionapi_request api_errormcp / auth / skill …

共通属性: session.id / user.email / organization.id / terminal.type / model …
✓ 「誰が・いつ・どのツールを・成否・トークン/コスト」はこれだけで取れる

※ コストは「API 単価で計算した推定値」。Pro/Max などの定額プランでは実際には課金されない(利用量の目安)

まず素の状態。メトリクスでトークン・コスト・セッション数。 イベントで user_prompt、tool_result、api_request など。 各レコードに user.email や session.id が付くので、誰がどのツールをいくらで使ったかはこれで分かります。 ただし「中身」、入力本文や Claude の回答は、デフォルトでは伏せられています。

★ フラグを開けると「中身」まで取れる

取りたいもの環境変数
ユーザ入力 本文OTEL_LOG_USER_PROMPTS=1
tool 入力引数 (bash cmd, path…)OTEL_LOG_TOOL_DETAILS=1
tool 入出力 コンテンツOTEL_LOG_TOOL_CONTENT=1 (traces必須)
Claude 出力本文・会話履歴全文OTEL_LOG_RAW_API_BODIES=1
分散トレース (interaction→tool)CLAUDE_CODE_ENHANCED_TELEMETRY_BETA=1

既定はすべて OFF(プライバシー)。有効化すると会話全文・bash コマンド・ファイル内容が残る → 取扱い注意

ここが山場。これらのフラグを開けると、入力本文、ツールの引数、ツールの入出力、 そして Claude の出力本文・会話履歴の全文、さらに分散トレースまで取れます。 デフォルト OFF なのはプライバシーのため。逆に言うと、開ければ監査に必要なものはほぼ全部取れる。 次のデモで実際に取れているところを見せます。

デモ: 実際に取れている

入力本文 → Claude 出力本文 → tool 引数 → トレース階層 が全部ローカルに

デモです。フル監査モードで claude を1回叩いて、 ① 入力本文、② Claude の出力本文、③ ツールの入力引数、④ Tempo のトレース階層、 の順に、ローカルに残ったものを表示しています。全部手元のファイルとバックエンドに入っています。

Grafana で可視化（GUI ウォークスルー）

ダッシュボード → ログ(Loki) → トレース(Tempo) を字幕付きで解説 / 詳細は presentation/gui-guide.md

可視化は Grafana。動画で順に解説しています。 まずダッシュボードで“量”の全体像、トークン・コスト・セッション数。 次に Loki でイベントの“中身”、行を開くと user.email や入力本文まで属性が見える。 最後に Tempo でトレース、1プロンプトの処理の流れと、ツールのどこで時間がかかったかが分解できます。 ログの trace_id からトレースへ、相互ジャンプも設定済みです。
